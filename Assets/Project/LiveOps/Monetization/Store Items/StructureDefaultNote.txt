===============================
STRUCTURE DEFAULT - CLEAN + MVVM
===============================

Mục tiêu:
- Tách biệt rõ trách nhiệm từng tầng
- Dễ mở rộng
- Dễ test
- Dễ thay SDK (Ads, IAP, Analytics…)
- Phù hợp production game


=================================
1. TỔNG QUAN KIẾN TRÚC
=================================

Luồng phụ thuộc:

Presentation  →  Application  →  Domain
Infrastructure → Domain
Installer → Tạo và kết nối tất cả

Nguyên tắc quan trọng:

- Domain KHÔNG phụ thuộc bất kỳ tầng nào
- Application KHÔNG phụ thuộc Unity
- Presentation KHÔNG chứa business logic
- Infrastructure có thể dùng SDK ngoài
- Installer là nơi duy nhất được phép "new"


=================================
2. DOMAIN (Tầng lõi)
=================================

Chứa:
- Entity
- Value Object
- Interface (Repository, Provider, Policy…)
- Business rule thuần logic

Không được:
- Dùng MonoBehaviour
- Dùng UnityEngine
- Gọi SDK ngoài

Ví dụ:
- CurrencyState
- IRewardPolicy
- IIAPRepository
- AdPolicy

Domain = Trái tim hệ thống


=================================
3. APPLICATION (UseCase / Service)
=================================

Chứa:
- UseCase
- Service
- Logic điều phối

Nhiệm vụ:
- Thực thi nghiệp vụ game
- Gọi Repository / Provider
- Update Domain

Không được:
- Truy cập UI
- Truy cập MonoBehaviour

Ví dụ:
- PurchaseUseCase
- CurrencyService
- ShowAdUseCase

Application = Não điều khiển


=================================
4. INFRASTRUCTURE (SDK / Save / Analytics)
=================================

Chứa:
- Triển khai cụ thể của interface trong Domain
- Tích hợp SDK bên thứ 3
- Save/Load
- Analytics
- Ads Provider
- IAP Provider

Ví dụ:
- EasySaveCurrencyRepository
- GleyIAPProvider
- FirebaseIAPAnalytics

Infrastructure có thể dùng Unity & SDK ngoài.

Infrastructure = Tầng thay thế được


=================================
5. PRESENTATION (UI Layer)
=================================

Chứa:
- View (MonoBehaviour)
- ViewModel (Reactive)
- Presenter

Nhiệm vụ:
- Nhận input người chơi
- Bind dữ liệu ra UI
- Gọi UseCase

Không được:
- Chứa business logic
- Gọi SDK trực tiếp
- Lưu dữ liệu

Presentation = Tầng giao tiếp với Unity


=================================
6. INSTALLER (Composition Root)
=================================

Nhiệm vụ:
- Tạo dependency graph
- Kết nối Domain + Application + Infrastructure + Presentation

Ví dụ:

Repository
   ↓
UseCase
   ↓
ViewModel
   ↓
Presenter

Installer là nơi duy nhất được phép new concrete class.


=================================
7. LUỒNG HOẠT ĐỘNG VÍ DỤ (IAP)
=================================

1. Player bấm mua
2. View gọi Presenter
3. Presenter gọi UseCase
4. UseCase gọi Provider
5. Provider trả kết quả
6. UseCase update Domain
7. ViewModel nhận event
8. UI cập nhật

UI không biết Gley là gì.
UseCase không biết MonoBehaviour là gì.
Domain không biết Unity là gì.


=================================
8. NHỮNG ĐIỀU CẤM
=================================

❌ Không gọi Firebase trực tiếp trong View
❌ Không lưu PlayerPrefs trong Domain
❌ Không new Repository trong Presenter
❌ Không để MonoBehaviour trong Domain
❌ Không để business logic trong View


=================================
9. KHI NÀO TẠO MODULE MỚI?
=================================

Khi hệ thống đủ lớn:

- CurrencyModule
- IAPModule
- AdsModule
- LiveOpsModule
- MissionModule

Mỗi module đều có:

Application
Domain
Infrastructure
Installer
Presentation


=================================
KẾT LUẬN
=================================

Cấu trúc này dùng cho:

- Game mobile production
- Idle tycoon
- Game LiveOps
- Game scale UA lớn
- Dự án nhiều lập trình viên

Đây không phải over-engineering.
Đây là kiến trúc chuẩn cho dự án dài hạn.
